## 二、我的逐条分析（第一性原则）

### 关于 #A「必须给结论」—— ✅ 完全认同，但要做"分层结论"

BPO 老大说得对：**招聘者要的是路由决策，不是阅读理解。**

但我认为需要区分两层：

| 层次 | 结论类型 | 置信度要求 | 举例 |
|------|---------|----------|------|
| **L1 硬性筛选**（事实型） | 二元判断：符合/不符合 | 高，可自动化 | 工作年限≥3年、期望薪资≤35K、到岗≤1个月 |
| **L2 能力评估**（判断型） | 三级路由：Proceed/Follow-up/Hold | 中，需证据支撑 | 技术深度、项目真实性、离职动机 |

> [!IMPORTANT]
> L1 是可以「零 token」完成的——从简历文本直接提取，做规则匹配。这才是真正的"高效路由"。
> L2 才需要 AI 对话 + 证据判断。两层分开，效率最高。

**产品决策建议**：报告页最顶部放「Decision Card（结论卡）」，一屏给出结论 + 3 条理由 + 证据入口。

---

### 关于 #B「标准要白盒」—— ⚠️ 核心分歧点，需要分场景讨论

BPO 老大建议"3 道题全对就 Proceed"。这个思路对 BPO 场景成立，但对我们的目标用户未必。

**分歧的本质：**

| 用户类型 | 有没有能力出题 | 适合的模式 |
|---------|-------------|----------|
| **BPO 专业招聘者** | ✅ 有丰富经验，能出精准筛选题 | 自定义题组（她的建议） |
| **业务方自招（我们的核心用户）** | ❌ 大概率不能，需要引导 | AI 推荐 + 可微调 |

**我的判断：两种模式都要支持，但 MVP 应该先做「AI 推荐」模式。**

理由：
1. 如果连产品总监（你）都觉得"想不出题"，那业务招聘者更想不出
2. AI 推荐有容错空间——推荐 3 题让用户确认，比让用户从零出题容易 10 倍
3. BPO 题组可以作为「高级功能」后续迭代，满足专业用户

**MVP 的 KSQ（Key Screening Questions）机制建议：**

```
1. AI 阅读 JD + 简历 → 自动生成 3 道 KSQ
2. 每道题附带：题目 + 考察意图 + 合格判定标准（rubric）
3. 招聘者可以：确认 / 替换 / 微调措辞
4. 面试后，AI 按 rubric 自动给出"满足/部分满足/不满足"
```

> [!NOTE]
> 关于你担心的"AI 精准度"问题——**rubric 的引入本身就是降低精准度要求的手段**。
> 不是让 AI 主观判断"这人行不行"，而是让 AI 对照 rubric 判断"回答有没有命中关键词/关键点"。
> 这是模式匹配，不是智力判断，LLM 是可以做好的。

---

### 关于 #C「先结论后证据」—— ✅ 完全认同，呈现顺序必须改

当前 `OrderDetailView` 的结构是：

```
候选人信息 → 五段式观察列表 → 建议追问 → AI 建议（底部）
```

应该改为：

```
1. Decision Card（结论卡）——一屏给完
   ├── 结论：Proceed / Follow-up / Hold
   ├── 置信度：85%（针对整体结论）
   ├── Top 3 理由（每条绑定证据锚点）
   └── 建议下一步（追问 / 约二面 / 放弃）

2. KSQ 回答卡——白盒验证区
   ├── Q1: [题目] → 满足 ✅ → [证据]
   ├── Q2: [题目] → 部分满足 ⚠️ → [证据]
   └── Q3: [题目] → 不满足 ❌ → [证据]

3. 五段式观察详情——证据库（可折叠）
   ├── 硬技能评估（置信度高）
   ├── 经验核实（置信度中）
   └── 软素质印象（标注：仅供参考）
```

> [!TIP]
> 这个结构也解决了你说的"信任问题"——招聘者先看结论，不信就展开看 KSQ 回答，还不信就点进五段式看原始证据。**信任是分层建立的。**

---

### 关于 #D「软素质高风险」—— ✅ 认同，但不是不做，是换个方式呈现

| 维度 | AI 可以做什么 | AI 不应该做什么 |
|------|------------|--------------|
| 沟通表达 | 记录"语速变化、停顿时长、逻辑断层" | 给出"沟通能力 85 分" |
| 逻辑思维 | 标记"回答前后矛盾、数据不一致" | 判断"逻辑思维强/弱" |
| 抗压能力 | 完全不可考察 | 任何评价 |

**产品决策建议**：
- 硬技能 + KSQ 回答 → 给明确"满足/不满足"判断
- 软素质 → 只提供**客观行为证据**，标注「仅供参考，建议人工复核」
- 在 Decision Card 的置信度中，如果结论主要依赖软素质证据，置信度自动降低

## 三、MVP 改进优先级排序

| 优先级 | 改动 | 复杂度 | 价值 | 依赖 |
|--------|------|--------|------|------|
| **P0** | 报告页顶部增加 Decision Card（结论卡） | 中 | 极高 | 无 |
| **P0** | 结论呈现顺序倒转：结论→理由→证据 | 中 | 极高 | Decision Card |
| **P1** | KSQ 机制（AI 推荐 3 题 + rubric + 判定） | 高 | 高 | AI prompt 工程 |
| **P1** | 软素质标注"仅供参考"标签 | 低 | 中 | 无 |
| **P2** | L1 硬性筛选自动化（薪资/年限/到岗） | 中 | 中 | 简历解析增强 |
| **P2** | 自定义题组（BPO 模式） | 中 | 高（特定客群） | KSQ 机制完成 |

## 五、待讨论的关键问题

1. **KSQ 的 3 道题从哪来？**
   - MVP 阶段：AI 根据 JD + 简历自动推荐，附带推荐理由
   - 后续：支持企业自定义题组
   
2. **rubric 的精度能满足需求吗？**
   - 建议先用"关键词命中 + 数据量化"作为判定标准，不做主观评价
   - 例：Q="管过多少人？" rubric="≥20人=满足，5-20人=部分满足，<5人=不满足"
   
3. **Decision Card 的结论是 AI 独立给出，还是需要招聘者确认 KSQ 后才出？**
   - 建议：AI 先给初步结论 → 招聘者可在 KSQ 卡片上 override
