## 一、产品定义

### 我们到底在做什么

一句话：**用人者上传简历 → AI 自动深度面试 → 生成"简历声称 vs 面试核实"对比报告 → 用人者 5 分钟做出决策。**

### 核心用户：用人者（不是 HR）

用人者 = 自己招人的创始人、业务 Leader、小团队负责人。他们没有专职 HR，自己看简历、自己面试、自己拍板。

**为什么不是 HR？**
- HR 本来就会做初筛，AI 只是帮他快一点——价值是"省时间"，不够强
- 用人者不会面试、不知道怎么识别注水——AI 是填补能力空白，价值是"避免招错人"
- "省时间"很难让人付费和传播，"避免招错人"可以

**用人者的核心情绪**：
- 愤怒：花 1 小时面了个注水的人
- 恐惧：请到表里不一的人，试用期才发现
- 无助：不知道该问什么、怎么识别真假
- 焦虑：岗位空缺影响业务，必须快速做出决策

### 冷启动用户：猎头 / 独立招聘顾问

**为什么是猎头而不是直接找用人者？**
- 用人者招聘需求低频（可能半年招一人），难以快速积累数据和验证留存
- 猎头高频使用（每周推多个候选人），能快速滚动飞轮
- 猎头极度需要"拿得出手"的专业交付物来给甲方邀功
- 猎头交付的报告自带品牌水印 → 甲方看到 → 反向被种草 → 经典 PLG 传播

### 一句话定位

**"别把时间浪费在'聊聊看'上。"**

为什么选这句：
- "聊聊看"是用人者最真实的场景——看完简历觉得"好像行"，约来聊聊，结果浪费 1 小时
- 直接命中决策焦虑，同时暗示了解决方案：不需要"聊聊看"了，AI 已经帮你深入聊过了

备选（对内功能引导可用）：
- 防御型："让注水简历，止步于初面之前"
- 赋能型："每份简历背后，AI 帮你找真相"

### 话术原则（B/C 端绝不混用）

| 面向 | 调性 | 示例 |
|------|------|------|
| B 端（用人者/猎头） | 防御、恐惧驱动 | "看穿简历包装"、"你的决策防线" |
| C 端（候选人） | 温和、尊重、无压力 | "一次简短的沟通"、"帮助双方更好地了解彼此" |

**为什么必须分开**：如果候选人发现产品本质是"抓他造假的工具"，面试完成率会断崖式下跌。B 端用恐惧拉注册，C 端用尊重保完成率。

## 二、产品体验设计：情绪曲线

### 核心问题

产品的验证价值来自 AI 面试（异步过程，约 25 分钟）。纯看简历发现不了任何问题。所以 Aha Moment 不可能是即时的。

**解法**：不是缩短等待，而是**把等待变成一段充满参与感的旅程，让报告到达时产生最大的情绪冲击**。

### 情绪曲线设计

```
上传简历
  → T+0:    【Mini Aha】"疑点雷达"立刻显示 3 个值得核实的点
  → T+1min: 邀约发出，"侦查启动"
  → T+X:    候选人打开链接 → 通知用人者
  → T+Y:    候选人开始面试 → "AI 正在深度核实..."
  → T+25min:【真正的 Aha】报告到达——"声称 vs 核实"对比揭晓
```

### 三个关键设计

**1. 疑点雷达（T+0 的 Mini Aha）**

Resume Agent 解析简历时，同时输出疑点：

```
扫描完成 · 发现 3 个值得深挖的点

1. 声称"带领12人团队"，但职级为"高级工程师"
   → 同行业同职级通常管2-4人，将在面试中核实

2. "营收增长300%"，任职周期仅7个月
   → 时间窗口与声称成果存在疑问，将追问数据来源

3. 算法竞赛经历（可信度较高，低优先级核实）
```

**为什么这是强 Aha**：用人者自己看简历大概率没发现这些问题。AI 发现了他没发现的东西——即时建立信任。

**2. 中间态通知（事实性，不预判）**

好的通知："张三已回答第 2/3 个核心问题，面试进行中..."
坏的通知："候选人回避了核心数据"（这是结论，留到报告揭晓更有冲击力）

原则：**好的悬疑电影不会在第二幕就告诉你凶手是谁。**

**3. 报告首屏 ="声称 vs 核实"对比**

```
张三 · 后端工程师 · 核实报告

  简历声称              AI 核实结果
  ─────────────────────────────────────
  带领 12 人团队   →   实际：2人直属 + 外包协调
  营收增长 300%    →   实际：部门整体数据，本人贡献不清
  5年后端经验      →   技术深度符合声称

  AI 建议：值得跟进，但需追问管理经历真实性

  [查看完整证据] [安排正式面试] [暂时搁置]
```

核心情绪：**不是打分，是抓包。** "抓包"带来的惊讶、确认、感激——是最容易引发分享的情绪。

## 三、增长引擎

### 飞轮 1：报告即传播物料（猎头 PLG）

```
猎头用 Ailin 生成报告 → 报告随简历交付给甲方
→ 甲方用人者看到报告 → "这个报告不错，比简历清楚多了"
→ 甲方反向被种草 → 自己也开始用 Ailin
```

执行要点：报告 PDF / 分享页底部必须有 "由 Ailin AI 生成 · ailin.cn" 水印 + 注册入口。每一次报告流转都是获客渠道。

### 飞轮 2：量化避免的损失（留存 + 口碑）

每月推送：
```
3月，Ailin 帮你：
  筛查了 12 份简历
  发现 4 份存在明显注水
  节省了约 4 小时无效面试时间
```

用人者会截图发朋友圈——这是他的工作效率，他的面子。

### 飞轮 3：候选人体验反向传播

面试结束后，候选人收到正向反馈（只展示 CONFIDENT 信号）：
> "感谢你的时间。你关于 XX 项目的描述非常清晰，这部分我们记录得很完整。"

候选人感觉："这家公司面试流程好专业" → 雇主品牌溢价 → 更多候选人愿意认真参与。

## 四、护城河战略

### 残酷的现实

一个"帮你分析简历 + 做 AI 面试"的 Chrome 插件，任何团队一个月内就能复制。工具层没有壁垒。

```
现在：蓝海
6个月后：竞品涌现，功能趋同
1年后：大厂入局（字节、腾讯、猎聘）
2年后：工具层彻底商品化
```

**结论：工具层只是入场券，不是终点。**

### 一个被否定的方向：C 端资产化（能力通行证）

曾经的设想：每次面试积累 Verified Tags → 候选人拿到"能力通行证" → 双边市场。

**为什么是伪命题（柠檬市场 / 逆向选择）**：
- 真正优秀的候选人被企业追着要，不需要"证明自己"
- 主动来"认证"的都是需要证明自己的弱势群体
- B 端用人者发现池子里都是弱势候选人 → 平台价值坍塌

这和二手车市场的逻辑完全一样。

### 真正的护城河：双层数据飞轮

**第一层：Global Loop（反欺诈知识图谱）— 服务所有用户**

```
企业 A 的面试发现"声称精通微服务，实际只做过单体应用"
→ 这个模式入库
→ 企业 B 收到类似简历时，疑点雷达标注：
  "此类经历声称在历史面试中有 60% 的不一致率"
```

纯 B 端网络效应：用的企业越多 → 每个企业得到的"疑点扫描"越精准。不依赖 C 端参与。

**第二层：Local Loop（品味对齐）— 服务单个用户**

招聘是极度非标的偏好游戏。同样"带 10 个人但 8 个是外包"：
- Leader A 愤怒于不诚信
- 创始人 B 觉得只要能打仗就行

系统通过用人者的隐性反馈（对 Observation 的反应、决策模式、KSQ 修改行为）逐步学习该用人者的偏好，做到"懂你的习惯"。

**大厂入局时的防御**：大厂产品是通用的，没有被这个 Leader "调教"过。高净值用户不会轻易迁移到从零开始的竞品。

### 护城河三阶段演进

```
Phase 1（现在）：工具层
  壁垒：无（接受现实，专注积累数据）
  核心指标：累计面试次数、面试完成率、报告分享率
        ↓ 关键转折：几千次真实面试
Phase 2（6-18月）：数据层
  Global：反欺诈知识图谱，疑点雷达越来越准
  Local：个性化对齐，系统越用越懂用人者
  壁垒：竞品没有你的面试数据
        ↓ 关键转折：量变引起质变
Phase 3（18月+）：行业基础设施
  零样本预判：遇到陌生简历，仅凭模式就能预判风险
  商业模式：不再是软件工具，而是"简历可信度的穆迪评级"
  壁垒：数据规模 + 时间壁垒，极高
```

**Phase 1 的唯一价值不是赚钱，是积累 Phase 2 的弹药。每一次面试都在建造护城河。**

### 定位不是二选一，是递进

讨论中曾纠结"专家能力民主化"（让不会面试的人也能精准筛人）和"情报前置"（见面前就知道值不值得见）两个方向。最终结论：**这不是选择题，是递进关系。**

```
Phase 1 定位：「情报前置」
  → "见面之前，先知道值不值得见"
  → 卖的是报告的即时价值

Phase 2 定位：「越用越准的招聘情报」
  → "你用得越多，它看人越准"
  → 卖的是系统的学习能力

Phase 3 定位：「行业级简历可信度基准」
  → "XX 行业已有 N 万次核实数据"
  → 卖的是数据规模带来的壁垒
```

每个 Phase 的定位自然承接上一个，不需要推翻重来。

## 五、当前代码的关键断裂点

> 以下问题不是 bug，是架构盲区。每一个都直接影响"报告是否可信"。
> **如果 AI 输出质量不过关，再精美的话术和增长策略都没有意义。**

| # | 断裂点 | 当前状态 | 后果 |
|---|--------|---------|------|
| 1 | Analysis Agent 看不到完整简历 | 只传了一行摘要 | "声称 vs 核实"对比是盲猜，CONTRADICTORY 信号不可信 |
| 2 | Interview Agent 话题切换 | 字符串匹配 `reply.includes('下一个')` | 面试卡死在同一个 KSQ 或过早结束 |
| 3 | LLM 输出无 schema 验证 | `JSON.parse()` 裸调用 | 一次格式错误 → 整条流水线崩溃 |
| 4 | 无用户反馈回路 | 用人者决策数据死在前端 | 系统永远不知道推荐是否正确，无法自我校准 |
| 5 | TagService 不持久化 | 提取了 tags 但没写入数据库 | 知识图谱的数据源不存在 |

**最严重的是第 1 条**：产品核心卖点是"声称 vs 核实"，但 Analysis Agent 实际上没有完整简历来做交叉比对。

**当前产品本质上是一个 Static LLM Wrapper**：第 1000 次面试和第 1 次面试使用完全相同的 Prompt，有完全相同的"智商"。系统不会因为用得更多而变得更聪明。

## 六、学习闭环：Compound Learning Loop

### 架构全景

```
                    ┌─────────────────────────────────┐
                    │                                 │
                    v                                 │
上传简历 → 疑点雷达 → AI 面试 → 分析报告 → 用人者决策 ─┘
    │           │         │          │          │
    v           v         v          v          v
 [简历特征库]  [疑点模式库]  [追问策略库]  [信号精度库]  [决策标签库]
    │           │         │          │          │
    └───────────┴─────────┴──────────┴──────────┘
                          │
                          v
              下一次面试的每个环节都更精准
```

### 五条反馈回路（按实现优先级排序）

**回路 1：决策标签（Phase 1 必做，成本极低）**
- 用人者点击 Proceed / Hold / Pass → 写入数据库
- 关联对应的 Observations + KSQ Results + AI 推荐
- 这是整个学习闭环的 ground truth

**回路 2：KSQ 有效性（Phase 1 埋点，Phase 2 分析）**
- 统计哪个 KSQ 产生了最多有价值的 Observations
- 高效的 KSQ 模式被优先推荐

**回路 3：疑点模式库（Phase 2 核心）**
- "声称 vs 核实"数据结构化入库
- 形成行业级别的疑点先验概率

**回路 4：追问策略（Phase 2-3）**
- 学习哪种追问方式最能让候选人暴露真实信息
- Interview Agent 的 prompt 动态调整

**回路 5：简历特征（Phase 3）**
- 学习哪些简历写法/模式与"注水"相关
- 需要大量数据，长期积累

### 隐性数据汲取机制

不要指望用人者填表单。在业务流里自然获取反馈：

**机制 A：Observation 反应手势**
- 报告的每个 Observation 旁边放 [在乎] [无所谓] [不准] 三个轻量按钮
- 用户在看报告时不知不觉完成了 RLHF 标注
- 积累 10-20 次后，该用人者的偏好画像形成
- 下次面试时注入 System Prompt："该用人者对管理经验包装容忍度高，对抗压能力零容忍"

**机制 B：KSQ 修改行为追踪**
- 用人者审核 KSQ 时的修改（删除/新增/调整评分标准）
- 隐式提取了该用人者的考察偏好
- 不需要额外交互，纯后台记录

## 七、路线图

### 总览

```
Phase 0 · AI 质量达标（1-2 周）
  修好核心断裂点，确保报告可信
        ↓
Phase 1 · MVP 上线 + 冷启动（第 3-8 周）
  文字面试全链路 + 疑点雷达 + 首批用户
        ↓
Phase 2 · 学习闭环 + 产品进化（第 3-6 月）
  系统越用越准，用户感知到进化
        ↓
Phase 3 · 数据壁垒 + 规模化（第 6-18 月）
  行业级别知识图谱，竞品无法复制
```

### Phase 0：AI 质量达标（上线前 · 1-2 周）

**目标**：让 AI 输出达到"用人者愿意基于它做决策"的可信度。

| # | 改动 | 涉及文件 | 工作量 |
|---|------|---------|--------|
| 0.1 | Analysis Agent 接收完整 DetailedResume | `analysisAgent.ts`, `orchestrator.ts` | 2h |
| 0.2 | Interview Agent 话题切换改为 LLM 结构化输出 | `interviewAgent.ts`, `interview.ts` | 3h |
| 0.3 | LLM 输出增加 Zod schema 验证 + 容错重试 | 所有 Agent | 4h |
| 0.4 | TagService 写入数据库 | `tagService.ts`, `orchestrator.ts` | 2h |
| 0.5 | 端到端测试：3-5 份真实简历 | 手动 | 4h |

**验收标准**：
- 上传真实简历 → 文字面试 10-15 轮 → 报告包含至少 1 条有说服力的信号
- Interview Agent 能自然地按 KSQ 推进、追问、切换话题
- LLM 格式错误不会导致流水线崩溃

### Phase 1：MVP 上线 + 冷启动（第 3-8 周）

**目标**：产品可用，获取首批 5-10 个猎头/用人者用户。

| # | 功能 | 优先级 |
|---|------|--------|
| 1.1 | 简历疑点雷达（T+0 Mini Aha） | P0 |
| 1.2 | 报告首屏改为"声称 vs 核实"对比 | P0 |
| 1.3 | 用人者决策落库（学习闭环 ground truth） | P0 |
| 1.4 | 报告品牌水印 + 一键分享 | P1 |
| 1.5 | 面试进度事实性通知 | P1 |
| 1.6 | 接通 Supabase + LLM API | P0（阻塞一切） |

**需要验证的三个核心假设**：

| 假设 | 成功标准 |
|------|---------|
| 候选人愿意接受 AI 面试 | 打开率 > 60%，完成率 > 40% |
| 报告对用人者有决策价值 | > 70% 的报告被阅读后有决策动作 |
| 猎头愿意使用并分享报告 | 每 10 份报告至少 3 份被分享 |

**MVP 阶段的核心指标**（不是月活，是数据积累）：
- 累计完成面试次数（目标 50+）
- 面试完成率（目标 > 40%）
- 报告分享次数（目标 15+）

**冷启动执行**：
1. 找 3-5 个猎头做种子用户（免费 1 个月，换取反馈）
2. 猎头交付给甲方的报告自带水印 → 甲方反向获客
3. 目标：第 1 个月末有 2-3 个甲方主动找过来

### Phase 2：学习闭环 + 产品进化（第 3-6 月）

**前提**：累计 200+ 次面试 + 100+ 条决策标签。

| # | 功能 | 对应的学习回路 |
|---|------|--------------|
| 2.1 | 决策标签分析 | 校准 Report Agent 推荐逻辑 |
| 2.2 | KSQ 有效性评分 | 优先推荐高效 KSQ 模板 |
| 2.3 | 疑点模式库 V1 | 行业级别的疑点先验概率 |
| 2.4 | 学习仪表盘 | 让用户感知到系统在进化 |
| 2.5 | STT 接入（讯飞/阿里云） | 语音面试上线 |
| 2.6 | Prompt 版本化 | 支持 A/B 测试 |

**成功指标**：
- 累计面试 500+ 次
- AI 推荐与用人者决策一致率 > 60%
- 用户月留存 > 40%

### Phase 3：数据壁垒 + 规模化（第 6-18 月）

| # | 功能 | 壁垒意义 |
|---|------|---------|
| 3.1 | 行业疑点基准 | 后来者需要同样时间积累 |
| 3.2 | 追问策略自动优化 | Prompt 自动进化 |
| 3.3 | 简历可信度预评分 | 不需要面试就能给出评分 |
| 3.4 | 多垂直行业扩展 | 技术岗 → 销售岗、管理岗 |
| 3.5 | 企业版（多用户协作） | 团队共享、统一标准 |

## 八、关键风险
